Корпоративный RAG-ассистент
============================

О проекте
---------
Веб-приложение для ответов на вопросы по внутренним документам компании (PDF, TXT, Markdown).
Текст документов режется на фрагменты, векторизуется и сохраняется в Qdrant. При вопросе
система ищет релевантные фрагменты и передаёт их в языковую модель вместе с историей диалога.

Задача — показать рабочий end-to-end сценарий: загрузка → индексация → диалог с источниками.

Стек
----
Backend: Python 3.12, FastAPI, Qdrant (qdrant-client), httpx.
Frontend: React (Vite), TypeScript, Tailwind CSS.
Деплой: Railway.

Как это устроено
----------------
• POST /upload — приём файла, извлечение текста, разбиение на чанки, эмбеддинги, upsert в Qdrant.
• POST /ask — RAG: поиск по коллекции, сборка промпта, вызов LLM (OpenRouter), возврат ответа и имён файлов-источников.
• GET/DELETE /documents — список и удаление документов; DELETE /documents — очистка всей коллекции.
• GET/DELETE /conversations — список диалогов, история, удаление (хранение в памяти процесса; для продакшена нужен Redis и т.п.).
• Статика SPA отдаётся из каталога static рядом с приложением (сборка Vite в Docker).

Переменные окружения (основные)
--------------------------------
OPENROUTER_API_KEY, OPENROUTER_MODEL — доступ к LLM через OpenRouter.
MISTRAL_API_KEY — опционально, для эмбеддингов Mistral; без ключа используется локальный fallback фиксированной размерности (удобно для разработки, для качества поиска лучше настроить API).
QDRANT_URL / QDRANT_API_KEY или QDRANT_HOST — подключение к Qdrant.

Планы на будущее
----------------
• Хранение диалогов в Redis или БД вместо памяти процесса (сейчас при рестарте контейнера на Railway история теряется).
• Аутентификация пользователей и разграничение доступа к документам по ролям.
• Метрики и логирование запросов (latency, ошибки LLM/Qdrant) для продакшена.
• Тесты API (pytest) и простой CI на push в main.
• Улучшение качества поиска: только эмбеддинги от провайдера в проде, тонкая настройка чанков и порогов релевантности.
• Опционально: отдельный воркер для тяжёлой индексации больших PDF, очередь задач.

Деплой на Railway
-----------------
Сервис собирается из Dockerfile в корне репозитория: этап Node собирает фронт, этап Python поднимает FastAPI и отдаёт статику.
В переменных окружения Railway задаются OPENROUTER_*, при необходимости MISTRAL_API_KEY, QDRANT_URL (или хост Qdrant как отдельный сервис).
Health: GET /health.
